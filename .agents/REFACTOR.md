# REFACTOR.md

## Зачем нужен этот файл

Этот документ фиксирует план рефакторинга исследовательского пайплайна в `notebooks/`,
чтобы в любой момент можно было заново стартовать задачу и быстро восстановить:

- что сейчас устроено неудачно;
- какой должна быть целевая структура ноутбуков;
- какие артефакты должны сохраняться на каждом этапе;
- какие модели логично обучать только на `top_pairs`, а какие можно запускать на всех рядах;
- как выстроить чистый поток от EDA к экспериментам и финальным submission.

## Текущее состояние

### Что есть в репозитории

- `notebooks/01_eda_baseline.ipynb`:
  EDA, подготовка `train_processed/test_processed`, временные сплиты, mean-baseline.
- `notebooks/02_sarima_solution.ipynb`:
  старый вариант SARIMA на уровне `(cluster, family)`.
- `notebooks/03_baseline_sarima.ipynb`:
  финальный SARIMA на уровне `(store_nbr, family)`.
- `notebooks/04_catboost.ipynb`:
  CatBoost с лаговыми признаками, Optuna, recursive forecast.

### Основные проблемы текущей структуры

1. В `01_eda_baseline.ipynb` смешаны несколько ролей:
   - подготовка данных;
   - EDA;
   - построение сплитов;
   - baseline;
   - часть инженерии признаков.

2. `02_sarima_solution.ipynb` и `03_baseline_sarima.ipynb` описывают две разные методологии SARIMA:
   - кластерный вариант;
   - store-level вариант.
   Для основной линии работы это создает путаницу.

3. `03_baseline_sarima.ipynb` и `04_catboost.ipynb` дублируют общий каркас:
   - загрузка данных;
   - восстановление `splits`;
   - выбор `top_pairs`;
   - фильтрация train/val;
   - расчет метрик.

4. Поток от EDA к экспериментам выглядит не как единый pipeline, а как набор самостоятельных ноутбуков.

5. В текущем CatBoost-пайплайне есть признаки и решения, которые нужно перепроверить методологически:
   - `sales_sum` выглядит как потенциальная утечка;
   - часть признаков есть в train/val, но отсутствует в `test_processed`;
   - фичи пересчитываются повторно внутри Optuna, что замедляет работу.

## Фактический масштаб задачи

- `data/train_processed.csv`: около `3,008,016` строк.
- Всего рядов `(store_nbr, family)`: `1782`.
- В `03_baseline_sarima.ipynb` используется `top_pairs`:
  `357` рядов, покрывающих примерно `88.6%` общего оборота.
- На одном сплите после фильтрации по `top_pairs`:
  - train: примерно `575k-597k` строк;
  - val: `5712` строк.

## Ключевой вывод по вычислительной сложности

### Почему SARIMA тяжелая

SARIMA тяжела не потому, что датасет в целом слишком большой, а потому что:

- строится много локальных моделей;
- на каждую серию идет отдельная подгонка;
- требуется подбор параметров `(p, d, q)` и сезонных параметров;
- итоговая стоимость растет почти как:
  `число рядов x число сплитов x стоимость подбора одной модели`.

Для `1782` рядов это слишком дорого.
Даже для `357` рядов из `top_pairs` обучение остается заметно тяжелым.

### Почему остальные модели должны быть проще

Для табличных моделей задача иная:

- строится один общий датасет признаков;
- модель обучается глобально по всем строкам;
- нет отдельного перебора модели для каждого ряда;
- для Ridge / Lasso / Decision Tree / Random Forest / CatBoost стоимость обучения
  намного ближе к классической supervised learning на таблице.

Практический вывод:

- SARIMA логично оставлять на `top_pairs` или еще более узком подмножестве.
- Линейные модели, дерево, Random Forest и CatBoost можно и нужно пробовать на всех рядах.

## Целевая структура ноутбуков

Ниже описана рекомендуемая структура после рефакторинга.

### Вариант структуры

1. `01_prepare_eda.ipynb`
2. `02_mean_baseline.ipynb`
3. `03_sarima_store_level.ipynb`
4. `04_tabular_baselines.ipynb`
5. `05_catboost.ipynb`
6. `06_compare_models.ipynb`

Дополнительно:

- `02_sarima_solution.ipynb` оставить как архивный exploratory notebook,
  но не использовать как часть основной линии исследования.

## Что должно быть в каждом ноутбуке

### 1. `01_prepare_eda.ipynb`

Цель:

- один раз подготовить данные и артефакты для всех остальных экспериментов.

Содержимое:

- загрузка raw data;
- merge таблиц;
- очистка;
- базовая типизация;
- EDA;
- построение:
  - `train_processed.csv`,
  - `test_processed.csv`,
  - `zero_sales.csv`,
  - `splits.json`,
  - `top_pairs` (сохранить отдельно в `data/experiment_info/`, например `top_pairs.csv`).

Важно:

- в этом ноутбуке не должно быть тяжелых модельных экспериментов;
- baseline можно оставить только как краткий reference, но лучше вынести отдельно.

### 2. `02_mean_baseline.ipynb`

Цель:

- зафиксировать самый простой и воспроизводимый baseline.

Содержимое:

- загрузка `train_processed.csv`, `zero_sales.csv`, `splits.json`;
- mean-baseline;
- метрики по сплитам;
- сохранение результатов в `data/experiment_info/`.

Желательно:

- считать baseline в двух вариантах:
  - на всех рядах;
  - на `top_pairs`.

Это позволит честно сравнивать baseline и с SARIMA, и с глобальными моделями.

### 3. `03_sarima_store_level.ipynb`

Цель:

- сохранить только одну финальную версию SARIMA:
  на уровне `(store_nbr, family)`.

Содержимое:

- загрузка `train_processed.csv`, `splits.json`, `top_pairs.csv`;
- фильтрация на `top_pairs`;
- обучение SARIMA;
- fallback для коротких/нулевых рядов;
- метрики;
- сохранение предсказаний и таблицы качества.

Важно:

- кластерный SARIMA сюда не переносить;
- если кластерный вариант нужен в тексте, он должен упоминаться как предварительный пилотный эксперимент.

### 4. `04_tabular_baselines.ipynb`

Цель:

- построить единый табличный pipeline для нетяжелых моделей.

Содержимое:

- общий feature engineering;
- единый train/val protocol по `splits`;
- модели:
  - Ridge;
  - Lasso при необходимости;
  - Decision Tree;
  - Random Forest.

Обязательное требование:

- все эти модели должны обучаться на одинаковом наборе признаков;
- все метрики должны считаться по одному и тому же протоколу;
- результаты должны сохраняться в `data/experiment_info/`.

### 5. `05_catboost.ipynb`

Цель:

- оставить только CatBoost и связанные с ним специфические решения:
  Optuna, recursive forecast, final training, submission.

Содержимое:

- загрузка готовых feature-инструментов;
- подбор параметров;
- обучение на сплитах;
- recursive forecast;
- финальное обучение;
- submission.

### 6. `06_compare_models.ipynb`

Цель:

- собрать все результаты в одну таблицу и подготовить финальный материал для главы 3.

Содержимое:

- загрузка метрик всех моделей;
- сводная таблица;
- ранжирование;
- анализ качества и вычислительной стоимости;
- заготовки таблиц/рисунков для ВКР.

## Методологические правки, которые нужно сделать

### 1. Убрать или перепроверить утечки

В первую очередь проверить CatBoost и будущие табличные модели.

Подозрительные признаки:

- `sales_sum`
  - сейчас это агрегат продаж по `(date, store_nbr)`;
  - он содержит информацию о текущем дне и, вероятно, использует сам target;
  - для табличной модели это выглядит как утечка;
  - по умолчанию убрать из признаков.

### 2. Проверить признаки, которых нет в тесте

В `train_processed.csv` есть признаки, которых нет в `test_processed.csv`,
например `transactions`, `dcoilwtico`, `sales_sum`.

Нужно принять одно из двух решений:

- либо не использовать такие признаки в моделях, для которых нужен финальный submission;
- либо научиться формировать/прогнозировать их на тестовом горизонте.

По умолчанию для чистоты исследования лучше первое:

- выделить безопасный набор признаков, доступный и в train, и в test;
- использовать именно его для моделей, для которых планируется финальный submission.

### 3. Унифицировать протокол сравнения

Нужно явно разделить два режима:

- `research mode`:
  - сравнение моделей на одинаковых сплитах;
  - при необходимости на `top_pairs`;
- `submission mode`:
  - обучение легких глобальных моделей на всех рядах;
  - формирование финального прогноза для test.

### 4. Отделить общую инженерию признаков от CatBoost-специфики

То, что подходит всем табличным моделям:

- календарные признаки;
- лаги;
- rolling statistics;
- безопасные exogenous features;
- train/val split logic.

То, что должно жить только в CatBoost-ноутбуке:

- Optuna;
- CatBoost Pool;
- cat feature indices;
- recursive multi-step forecast;
- final fit + submission.

## Отдельная ветка: обучение на всех рядах и финальные submission

Это обязательная часть рефакторинга.

### Общая идея

SARIMA:

- не масштабировать на все `1782` рядов;
- оставить как тяжелый статистический baseline на `top_pairs`.

Нетяжелые модели:

- обучать на всех рядах;
- делать полноценный final submission.

### Какие модели считать нетяжелыми

- Mean baseline
- Ridge
- Lasso
- Decision Tree
- Random Forest
- CatBoost

Замечание:

- Random Forest тяжелее Ridge и дерева, но все равно намного реалистичнее для полного прогона,
  чем SARIMA с перебором параметров для каждой серии.

### Что нужно реализовать для каждой нетяжелой модели

1. Режим `cv on splits`
   - одинаковые сплиты;
   - одинаковые метрики;
   - сохранение результатов в `data/experiment_info/`.

2. Режим `fit on full train`
   - обучение на всех доступных train-данных;
   - использование только тех признаков, которые допустимы для реального test horizon.

3. Режим `predict test`
   - прогноз на `data/test_processed.csv`;
   - формирование `submission_*.csv`.

### Что нужно сохранить по итогам

Для каждой модели:

- файл метрик по сплитам;
- файл со средними метриками;
- при необходимости файл с out-of-fold / validation prediction;
- финальный `submission_*.csv`.

## Предлагаемая логика исследований после рефакторинга

### Этап 1. Подготовка

- привести в порядок `01_prepare_eda.ipynb`;
- сохранить все промежуточные артефакты;
- сохранить `top_pairs.csv`.

### Этап 2. Базовые модели

- mean-baseline:
  - CV на всех рядах;
  - CV на `top_pairs`;
  - final submission.

### Этап 3. SARIMA

- только `top_pairs`;
- только CV / сравнительный анализ;
- без попытки строить final submission на всех рядах.

### Этап 4. Табличные baseline-модели

- единый feature set;
- Ridge / Lasso / Decision Tree / Random Forest;
- CV на всех рядах;
- при желании дополнительно CV на `top_pairs` для прямого сравнения с SARIMA;
- final submission на всех рядах.

### Этап 5. CatBoost

- CV на `top_pairs` для прямого сравнения с SARIMA;
- затем CV на всех рядах;
- затем final submission.

### Этап 6. Сравнение

Собрать две сводные таблицы:

1. `SARIMA-comparable table`
   - mean-baseline (`top_pairs`);
   - SARIMA (`top_pairs`);
   - Ridge (`top_pairs`, если посчитан);
   - Random Forest (`top_pairs`, если посчитан);
   - CatBoost (`top_pairs`).

2. `Full-data table`
   - mean-baseline (all series);
   - Ridge;
   - Lasso;
   - Decision Tree;
   - Random Forest;
   - CatBoost.

Это снимет методологическую претензию о том, что одни модели сравниваются на одних данных,
а другие на других.

## Что особенно важно не забыть

1. Не использовать признаки с возможной утечкой.
2. Не использовать признаки, отсутствующие на тестовом горизонте,
   если модель потом идет в final submission.
3. Для SARIMA и CatBoost зафиксировать, сравниваются ли они:
   - на `top_pairs`,
   - или на всех рядах.
4. Для нетяжелых моделей обязательно сделать отдельный полный прогон на всех рядах.
5. Все метрики и submission сохранять централизованно в `data/experiment_info/`
   и `data/`.

## Короткий итог

Целевое состояние:

- `EDA -> артефакты -> baseline -> SARIMA -> tabular baselines -> CatBoost -> compare`

Главная идея:

- SARIMA оставить как тяжелый, но полезный статистический reference на `top_pairs`;
- все глобальные и нетяжелые модели довести до полного прогона на всех рядах
  и до финального `submission`.
